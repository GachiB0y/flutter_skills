# Project Structure & Monorepo / Структура проекта и монорепозиторий

## Monorepo Approach / Подход к монорепозиторию

Монорепозиторий имеет смысл, если вы в одиночку или вдвоём пилите и бэкенд, и фронт. Когда команда разрастается и фронтенд/бэкенд пилятся отдельно -- это уже не самая удобная история. В таком случае фронтенд-часть выделяется в отдельную папку (например, `frontend/`), а всё остальное не трогается.

## pubspec.yaml -- первое, что нужно открыть

Когда приходишь на проект, первым делом открывай `pubspec.yaml`, чтобы понять:
- С чем предстоит работать
- Какие зависимости используются
- Как устроен проект

### Workspaces (локальные пакеты)

Используются workspaces -- относительно новая фича Dart/Flutter, которая позволяет проще делать локальные пакеты:

```yaml
# pubspec.yaml
workspace:
  - packages/localization
  - packages/ui
```

В каталоге `packages/` вынесены:
- **Локализация** -- отдельный пакет
- **UI Kit** -- отдельный пакет

Потенциально имеет смысл вынести и базу данных в отдельный пакет.

### Выбор зависимостей -- минимализм

Принцип: минимум зависимостей. На проекте было зависимостей раза в три больше при всего одном экране. Почти все были выкинуты. Большинство оставшихся зависимостей -- от самого Flutter SDK, остальная часть -- Firebase.

Ключевые зависимости:
- **State management**: собственный пакет `control` (продвинутые ChangeNotifier'ы)
- **HTTP-клиент**: стандартный `http` (изначально был Dio, полностью переписано)
- **База данных**: `Drift` (SQLite) + `SharedPreferences`
- **SharedPreferences под веб**, Drift -- только под нативные платформы (WASM-сборка SQLite слишком тяжёлая)

### Anti-pattern: слишком много зависимостей

> "Зависимостей было раза в три больше, при том что там был всего один экран. Я почти всё выкинул."

## UI Kit as Local Package / UI Kit как локальный пакет

### Зачем выносить UI Kit в отдельный пакет

Когда вы верстаете, вам нужно быстро проверять компоненты. Без UI Kit процесс выглядит так:
1. Запускаете приложение
2. Авторизуетесь
3. Переходите на нужный экран
4. 100 раз нажимаете hot restart
5. Не можете нормально проверить адаптивность

### Правильный подход: отдельное example-приложение

```
packages/
  ui/
    lib/           # Компоненты UI Kit
    example/       # Отдельное приложение для проверки компонентов
      pubspec.yaml # Собственный pubspec
```

**Example-приложение** собирается под все платформы (особенно Windows/macOS/Linux), даже если основное приложение только под Android/iOS. Это позволяет:
- Верстать компоненты отдельно, без запуска основного приложения
- Проверять адаптивность -- просто сжимаете окно
- Видеть все иконки, типографику, шейпы в одном месте
- Делать Ctrl+S и сразу видеть результат

```dart
// Пример: отдельный экран в UI Kit example
// для проверки всех компонентов
class ComponentShowcase extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return ListView(
      children: [
        // Все иконки
        // Пузырьки чата
        // Пин-код
        // Типографика
        // Шейдеры
      ],
    );
  }
}
```

### Anti-pattern: выносить каждую фичу в локальный пакет

> "Есть заблуждения и попытки подходов переносить всё в локальные пакеты, прямо отдельную каждую фичу в пакет выносить. Это хреновый подход."

В локальные пакеты выносите только:
- Вещи, которые изменяются **независимо**
- **UI Kit** -- компоненты, типографику, иконки

Не выносите в локальные пакеты отдельные фичи.

## Верстка компонентами, а не магическими числами

### Anti-pattern: магические числа и константы

```dart
// ПЛОХО -- магическое число
Padding(padding: EdgeInsets.all(16))

// ПЛОХО -- вынесенное магическое число (всё ещё магическое!)
Padding(padding: EdgeInsets.all(UIConstants.defaultPadding))

// ПЛОХО -- файл с константами
class UIConstants {
  static const smallSize = 12.0;
  static const defaultPadding = 16.0;
}
```

> "Люди думают, что если они вынесут число в переменную, оно перестанет быть магическим. Ни хрена подобного."

### Правильный подход: компоненты

Верстайте компонентами. Дизайнеры тоже должны верстать компонентами:

```dart
// ХОРОШО -- компонент текста
class AppText extends StatelessWidget {
  // Инкапсулирует типографику, шрифты, стили
}

// ХОРОШО -- компонент кнопки
class AppButton extends StatelessWidget {
  // Инкапсулирует размеры, скругления, отступы
}
```

Тема (`ThemeData`, цвета, типографика) используется **внутри компонентов UI Kit**, а не напрямую в экранах приложения.

## Key Takeaways / Ключевые выводы

1. Первым делом открывай `pubspec.yaml` на новом проекте
2. Используй workspaces для локальных пакетов
3. Минимизируй зависимости -- большинство можно выкинуть
4. UI Kit выноси в отдельный пакет с example-приложением
5. Не выноси каждую фичу в отдельный пакет
6. Верстай компонентами, не магическими числами
7. SharedPreferences для веба, Drift/SQLite для нативных платформ
