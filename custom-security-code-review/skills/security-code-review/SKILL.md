# Security Code Review

> Скилл для ручного аудита безопасности кода. Проверяет 7 направлений, записывает все находки в файл `SECURITY_REPORT.md` с приоритизацией и конкретными фиксами. Не применяет фиксы автоматически — только репортит.

## Процесс работы

### 1. Определи стек проекта

Перед началом аудита изучи проект и определи:

- **Язык и фреймворк** — Next.js, Flutter, Django, Go, Rails, Spring и т.д.
- **База данных** — PostgreSQL + Supabase, MySQL, MongoDB, Firebase, SQLite
- **Auth-провайдер** — свой, Supabase Auth, Firebase Auth, NextAuth, Clerk
- **Платёжная система** — Stripe, ЮKassa, PayPal (если есть)
- **AI-интеграции** — Anthropic, OpenAI (если есть)
- **Деплой** — Vercel, Docker, VPS, Cloud Run

Это определяет, какие проверки релевантны. Например, RLS-политики проверяются только для Supabase/PostgreSQL, а `dangerouslySetInnerHTML` — только для React.

### 2. Спроси режим

- **Полный аудит** — все 7 направлений последовательно. Для нового проекта или перед продакшн-деплоем.
- **Целевая проверка** — конкретные направления по выбору. Например, "проверь только авторизацию и секреты".

Если пользователь не уверен — предложи полный аудит.

### 3. Создай файл отчёта

В корне проекта создай `SECURITY_REPORT.md`. Все находки пиши туда. Структура файла:

```markdown
# Security Report

**Проект:** [название]
**Дата:** [дата]
**Стек:** [стек]
**Режим:** полный аудит / целевая проверка

## Сводка

| Уровень | Количество |
|---------|-----------|
| КРИТИЧНО | 0 |
| ВАЖНО | 0 |
| РЕКОМЕНДАЦИЯ | 0 |

## Топ-3 приоритетных фикса

1. ...
2. ...
3. ...

---

## 1. Аутентификация
[находки]

## 2. Авторизация и IDOR
[находки]

...
```

### 4. Формат каждой находки

```markdown
### [SEC-001] Название проблемы

- **Уровень:** КРИТИЧНО / ВАЖНО / РЕКОМЕНДАЦИЯ
- **Файл:** `путь/к/файлу.ext:строка`
- **Проблема:** что не так
- **Риск:** сценарий атаки — что конкретно может сделать атакующий
- **Фикс:**

\`\`\`lang
// конкретный код исправления
\`\`\`
```

Нумерация сквозная: SEC-001, SEC-002, ... — чтобы пользователь мог ссылаться на конкретные находки при пошаговом исправлении.

Уровни:
- **КРИТИЧНО** — эксплуатируется прямо сейчас без дополнительных условий (утечка ключа в коде, SQL injection, IDOR без проверки владельца)
- **ВАЖНО** — серьёзный риск, но нужны условия (нет rate limiting на логин — нужен бот для эксплуатации)
- **РЕКОМЕНДАЦИЯ** — best practice, отсутствие которого повышает риск в будущем (нет HSTS, нет CSP)

### 5. Проходи направления последовательно

После каждого направления:
- Запиши находки в `SECURITY_REPORT.md`
- Дай пользователю краткую сводку в чате: "Направление 1 (Аутентификация): 0 КРИТИЧНО, 1 ВАЖНО, 2 РЕКОМЕНДАЦИИ"
- Переходи к следующему направлению

### 6. После всех направлений

Обнови сводку в начале файла (общее количество, топ-3 фиксов) и сообщи пользователю: "Отчёт записан в `SECURITY_REPORT.md`. Рекомендую начать с SEC-001, SEC-003, SEC-007 (критичные)."

## Направление 1: Аутентификация

Аутентификация — первая линия обороны. Если она дырявая, всё остальное бессмысленно: атакующий входит как легитимный пользователь и обходит все остальные проверки.

### Чеклист

1. **Хеширование паролей** — bcrypt (cost >= 10) / argon2 / scrypt. Красные флаги: md5, sha1, sha256 без salt, plaintext.
2. **Сессии** — имеют TTL, инвалидируются при логауте. Refresh-токены ротируются.
3. **Email-верификация** — обязательна перед активацией аккаунта (если применимо).
4. **Токены сброса пароля** — одноразовые, истекают через 1 час максимум.
5. **Rate limiting на логин** — макс. 5 попыток/мин на IP или аккаунт.
6. **Auth-секреты на клиенте** — JWT_SECRET, SERVICE_ROLE_KEY, session secret не должны попадать в клиентский код.
7. **OAuth-редиректы** — callback URL валидируется по whitelist (нет open redirect).
8. **Middleware / Guard** — все защищённые маршруты проверяют сессию.

### Где искать (по стекам)

| Стек | Файлы для проверки |
|------|-------------------|
| Next.js | `middleware.ts`, `app/api/auth/`, `next-auth` конфиг |
| Flutter | `lib/**/auth`, `lib/**/interceptor`, `lib/**/token` |
| Django | `settings.py` (AUTH_PASSWORD_VALIDATORS), `views.py` (login), `middleware.py` |
| Express/Fastify | `routes/auth.*`, `middleware/auth.*`, `passport` конфиг |
| Supabase | Dashboard → Auth settings, `supabase/config.toml` |

## Направление 2: Авторизация и IDOR

IDOR (Insecure Direct Object Reference) — атакующий подменяет ID в запросе (`?id=123` → `?id=456`) и получает чужие данные. Каждый запрос к данным должен проверять: "этот ресурс принадлежит текущему пользователю?"

### Чеклист

1. **Эндпоинты с ID из URL/body/query** — перед возвратом данных проверяется ownership.
2. **Запросы к БД** — фильтр по текущему пользователю:
   - SQL: `WHERE user_id = $current_user_id`
   - Supabase: RLS-политика с `auth.uid()`
   - MongoDB: `{ userId: currentUser._id }`
   - ORM: `.where('userId', user.id)` или scope
3. **UPDATE/DELETE** — принадлежность записи проверяется до операции, не после.
4. **Публичные роуты** — только те, что действительно должны быть публичными. Остальные закрыты auth middleware.
5. **RLS-политики (PostgreSQL/Supabase)** — включены на КАЖДОЙ таблице с данными пользователей, покрывают все 4 операции (SELECT, INSERT, UPDATE, DELETE).
6. **Server Actions / API handlers** — первая строка: получить текущего пользователя и проверить права.

### Мысленный тест

Для каждого эндпоинта спроси: "Что произойдёт, если пользователь A подставит ID пользователя B?" Если ответ — "получит/изменит данные B" — это КРИТИЧНО.

## Направление 3: Валидация ввода

Любое место, где внешние данные попадают в систему — форма, API, файл, URL-параметр, webhook — это точка атаки. SQL-инъекции, XSS, command injection начинаются с невалидированного ввода.

### Чеклист

1. **Серверная валидация** — каждая точка входа имеет схему валидации:
   - JS/TS: Zod, Joi, class-validator, io-ts
   - Python: Pydantic, marshmallow, Django forms
   - Dart/Flutter: проверка на сервере (не только виджет-валидация)
   - Go: validator, custom middleware
2. **SQL-запросы** — параметризованные, не конкатенация строк. Красный флаг: `"SELECT * FROM users WHERE id = " + userId`.
3. **HTML-вывод** — пользовательский контент экранируется:
   - React: нет `dangerouslySetInnerHTML` с сырыми данными
   - Vue: нет `v-html` с сырыми данными
   - Шаблоны: auto-escaping включён
4. **Загрузка файлов** — проверка MIME-типа, размера, расширения. Исполняемые файлы заблокированы.
5. **Redirect URL** — валидируется по whitelist (нет open redirect через `?redirect=https://evil.com`).
6. **JSON-поля** — имеют строгую схему, не принимают произвольную структуру.
7. **Числовые параметры** — `limit`, `offset`, `page`, `size` имеют верхнюю границу.

## Направление 4: Секреты и API-ключи

Утечка одного ключа = полный доступ к БД, списание средств с AI API, доступ к платёжной системе. Атакующий просто открывает DevTools в браузере или смотрит публичный репозиторий.

### Чеклист

1. **Нет хардкода** — API-ключи, database URLs, service keys не в исходном коде.
2. **`.env` в `.gitignore`** — все файлы с секретами исключены из git.
3. **`.env.example`** — содержит только плейсхолдеры (`DB_URL=postgresql://...`), не реальные значения.
4. **Серверные ключи не на клиенте:**
   - Supabase: `SUPABASE_SERVICE_ROLE_KEY` только на сервере (он обходит ВСЕ RLS-политики)
   - Stripe/ЮKassa: `secret_key` только на сервере, на клиенте — `publishable_key` / `shopId`
   - AI: `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` — только на сервере
5. **Публичные env-переменные** — `NEXT_PUBLIC_*`, `VITE_*`, `FLUTTER_*` не содержат секретов (всё с этим префиксом видно пользователю).
6. **Git-история** — не содержит ранее закоммиченных секретов.
7. **Хранение токенов** — auth-токены в httpOnly cookies (web) или secure storage (mobile), не в localStorage/sessionStorage.

### Как искать в коде

Ищи паттерны: `sk_live`, `sk_test`, `apikey`, `api_key`, `secret`, `password`, `token`, `DATABASE_URL`, `PRIVATE_KEY` в исходных файлах (исключая `node_modules`, `.env.example`, тесты с моками).

## Направление 5: Rate Limiting и защита от ботов

Без rate limiting один скрипт может: перебрать все пароли, исчерпать лимит AI API за минуты, скрейпить весь каталог, создать тысячи фейковых аккаунтов.

### Чеклист

1. **Login** — макс. 5 попыток/мин на IP.
2. **Регистрация** — макс. 3 аккаунта/час на IP.
3. **AI-генерация** — лимиты по плану подписки.
4. **API общий** — макс. 100 запросов/мин на пользователя (или по проекту).
5. **ID не перебираемы** — UUID вместо последовательных числовых ID.
6. **CAPTCHA** — на публичных формах (регистрация, контакт, обратная связь).
7. **Вебхуки** — проверка подписи (HMAC SHA-256 для Stripe/ЮKassa, secret token для Telegram).
8. **Body size limit** — ограничение размера запроса (1MB API, 10MB загрузка файлов — или по проекту).

### Где искать

- Middleware: rate limiter, throttle, guard
- Конфигурация веб-сервера: nginx `limit_req`, Caddy rate_limit
- Пакеты: `@upstash/ratelimit`, `express-rate-limit`, `django-ratelimit`, `shelf_rate_limiter`

## Направление 6: Деплой и мониторинг

Код может быть идеален, но если деплой открывает БД в интернет или отключает HTTPS — безопасности нет.

### Чеклист

1. **HTTPS** — принудительный (редирект HTTP→HTTPS, HSTS-заголовок).
2. **Секреты деплоя** — в Environment Variables / Vault, не в коде и не в Docker image.
3. **БД закрыта** — прямой доступ к базе из публичного интернета заблокирован.
4. **CORS** — строго: конкретные домены, не wildcard `'*'` на продакшне.
5. **Security-заголовки** — `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Content-Security-Policy`.
6. **Логирование** — auth-попытки (успех/неудача), API-ошибки (4xx, 5xx), подозрительные паттерны (много 401/403 с одного IP).
7. **Мониторинг** — аномалии: резкий рост запросов, необычные паттерны доступа.
8. **Бэкапы БД** — настроены, регулярные, проверены восстановлением.
9. **Preview/staging** — не имеют доступа к продакшн БД и продакшн секретам.

### Где искать (по стекам)

| Стек | Файлы для проверки |
|------|-------------------|
| Vercel | `vercel.json`, `next.config.js` (headers) |
| Docker | `Dockerfile`, `docker-compose.yml`, `.dockerignore` |
| Nginx | `nginx.conf`, `sites-available/*` |
| Flutter (mobile) | `AndroidManifest.xml` (permissions), `Info.plist`, network security config |
| Supabase | Dashboard → Database settings, API settings |

## Направление 7: Зависимости и edge cases

Уязвимая зависимость — это дыра, которую ты не писал, но за которую отвечаешь.

### Чеклист

1. **Аудит зависимостей:**
   ```bash
   # JavaScript/TypeScript
   npm audit

   # Dart/Flutter
   flutter pub outdated
   dart pub deps

   # Python
   pip audit
   safety check

   # Go
   govulncheck ./...

   # Ruby
   bundle audit check --update
   ```
2. **Нет известных CVE** в прямых зависимостях.
3. **Lock-файл** актуален и закоммичен (`package-lock.json`, `pubspec.lock`, `Pipfile.lock`).
4. **Edge cases:**
   - Что если внешний сервис (AI API, платёжная система) недоступен?
   - Что если webhook придёт дважды (идемпотентность)?
   - Что если пользователь отправит запрос с истёкшим токеном?

## Что этот скилл НЕ проверяет

- Инфраструктурные уязвимости (конфигурация серверов, firewall-правила)
- Физическая безопасность
- Social engineering
- DDoS-устойчивость на уровне инфраструктуры
- Бизнес-логика (только технические уязвимости)

Скилл фокусируется на коде и конфигурации, которые разработчик контролирует напрямую.
