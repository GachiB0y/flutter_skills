# Project Structure & Monorepo / Структура проекта и монорепозиторий

## Monorepo Approach / Подход к монорепозиторию

Используй монорепозиторий, если ты в одиночку или вдвоём пилишь и бэкенд, и фронт. Когда команда разрастается и фронтенд/бэкенд пилятся отдельно -- выделяй фронтенд-часть в отдельную папку (например, `frontend/`), а всё остальное не трогай.

## pubspec.yaml -- первое, что нужно открыть

Первым делом открывай `pubspec.yaml`, чтобы понять:
- С чем предстоит работать
- Какие зависимости используются
- Как устроен проект

### Workspaces (локальные пакеты)

Используй workspaces -- относительно новую фичу Dart/Flutter, которая позволяет проще делать локальные пакеты:

```yaml
# pubspec.yaml
workspace:
  - packages/localization
  - packages/ui
```

В каталоге `packages/` выноси:
- **Локализация** -- отдельный пакет
- **UI Kit** -- отдельный пакет

Потенциально имеет смысл вынести и базу данных в отдельный пакет.

### Выбор зависимостей -- минимализм

Держи минимум зависимостей. Большинство зависимостей выкидывай -- оставляй только необходимые.

Ключевые зависимости:
- **State management**: собственный пакет `control` (продвинутые ChangeNotifier'ы)
- **HTTP-клиент**: стандартный `http` (изначально был Dio, полностью переписано)
- **База данных**: `Drift` (SQLite) + `SharedPreferences`
- **SharedPreferences под веб**, Drift -- только под нативные платформы (WASM-сборка SQLite слишком тяжёлая)

### Anti-pattern: слишком много зависимостей

> "Зависимостей было раза в три больше, при том что там был всего один экран. Я почти всё выкинул."

## UI Kit as Local Package / UI Kit как локальный пакет

### Зачем выносить UI Kit в отдельный пакет

Когда верстаешь, тебе нужно быстро проверять компоненты. Без UI Kit процесс выглядит так:
1. Запускаешь приложение
2. Авторизуешься
3. Переходишь на нужный экран
4. 100 раз нажимаешь hot restart
5. Не можешь нормально проверить адаптивность

### Правильный подход: отдельное example-приложение

```
packages/
  ui/
    lib/           # Компоненты UI Kit
    example/       # Отдельное приложение для проверки компонентов
      pubspec.yaml # Собственный pubspec
```

**Example-приложение** собирай под все платформы (особенно Windows/macOS/Linux), даже если основное приложение только под Android/iOS. Это позволяет:
- Верстать компоненты отдельно, без запуска основного приложения
- Проверять адаптивность -- просто сжимаешь окно
- Видеть все иконки, типографику, шейпы в одном месте
- Делать Ctrl+S и сразу видеть результат

```dart
// Пример: отдельный экран в UI Kit example
// для проверки всех компонентов
class ComponentShowcase extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return ListView(
      children: [
        // Все иконки
        // Пузырьки чата
        // Пин-код
        // Типографика
        // Шейдеры
      ],
    );
  }
}
```

### Anti-pattern: выносить каждую фичу в локальный пакет

> "Есть заблуждения и попытки подходов переносить всё в локальные пакеты, прямо отдельную каждую фичу в пакет выносить. Это хреновый подход."

Выноси в локальные пакеты только:
- Вещи, которые изменяются **независимо**
- **UI Kit** -- компоненты, типографику, иконки

Не выноси в локальные пакеты отдельные фичи.

## Key Takeaways / Ключевые выводы

1. Первым делом открывай `pubspec.yaml` на новом проекте
2. Используй workspaces для локальных пакетов
3. Минимизируй зависимости -- большинство можно выкинуть
4. UI Kit выноси в отдельный пакет с example-приложением
5. Не выноси каждую фичу в отдельный пакет
6. SharedPreferences для веба, Drift/SQLite для нативных платформ
